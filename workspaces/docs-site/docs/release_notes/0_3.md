# Release 0.3

Incan 0.3 is the current development release. It picks up after the `0.2` line, which made the language surface more explicit around stdlib imports, Rust interop, library manifests, module state, and call-site generics.

The first user-facing work in `0.3` is about tightening tooling contracts rather than adding a new language surface. In particular, `incan fmt` now has a citeable vertical-spacing contract through RFC 053, so formatter output is less dependent on local heuristics and more predictable across CLI, editor, and library entry points.

This page will grow as more `0.3` work lands. If you are looking for the shipped `0.2` story, start with [Release 0.2](0_2.md).

For the current formatter contract, start with [Formatting](../tooling/how-to/formatting.md) and [RFC 053].

## What 0.3 is about

The `0.2` line made Incan's module, stdlib, and Rust interop boundaries much clearer. `0.3` continues from that baseline with a stronger emphasis on predictable generated output, contributor ergonomics, and the gaps that remain visible when real projects keep using the compiler day to day.

The release is still early, but the initial direction is already visible:

- formatter output should be governed by explicit contracts, not scattered newline decisions
- user-facing tooling behavior should match the docs closely enough that CI and editor integrations can rely on it
- work that was intentionally deferred from `0.2`, especially broader ownership and generated-runtime ergonomics, remains the main follow-up area

## Migrating from 0.2

There are no required source migrations for the first `0.3` formatter work.

Projects that gate on `incan fmt --check` should expect one-time vertical-spacing diffs when adopting a formatter that implements RFC 053. Those diffs are intentional: top-level `def` / `model` / `type`-like declarations get exactly two blank lines around them, following body-bearing members inside type bodies get exactly one blank line, and other same-scope transitions stay in the zero-or-one bucket.

## Major additions

### RFC 053 formatter vertical-spacing contract

`incan fmt` now follows RFC 053's three-bucket vertical-spacing model:

- **Exactly two blank lines** around top-level `def`, `class`, `model`, `trait`, `enum`, `type`, `newtype`, and `rusttype` declarations
- **Exactly one blank line** before a following body-bearing member inside a type body
- **At most one blank line** everywhere else, including import runs, adjacent constants/statics, ordinary statement blocks, and transitions involving module docstrings when no top-level spaced declaration is involved

The formatter also normalizes repeated empty lines inside docstring payloads, keeps abstract trait methods tight until a following default/body-bearing method, and treats stand-alone comments as leading or trailing bundles rather than allowing comments to satisfy blank-line quotas on their own.

The same spacing contract applies through the CLI and the library formatter API. `FormatConfig` still controls ordinary formatting options such as indentation and line length, but RFC 053 vertical-spacing buckets are not configurable.

See also: [Formatting](../tooling/how-to/formatting.md), [RFC 053].

## Detailed inventory

The sections above are the release story. The list below is the detailed inventory of language, compiler, runtime, tooling, and docs changes that have landed for `0.3`, grouped roughly by theme rather than by the order work happened to land.

### Tooling and formatter

- **Compiler/Tooling**: RFC 053’s vertical-spacing contract is now reflected in `incan fmt`: top-level `def` / `model` / `type`-like declarations keep two blank lines around them, adjacent constants/statics stay grouped unless they border one of those declarations, trait abstract methods stay tight until a following body-bearing member, docstring interiors collapse repeated empty lines, and stand-alone comments attach as leading/trailing bundles instead of accidentally satisfying blank-line quotas (#336, RFC 053).

### Versioning and release track

- **Project metadata**: Workspace package metadata, Cargo lock entries, stdlib version-check docs, and checked-in pro example lockfiles now identify the development line as `0.3.0-dev.0`.

## Known limitations (0.3)

- `incan fmt` remains intentionally conservative on broader wrapping and may leave ordinary signatures, constructor calls, and nested expressions beyond the documented 120-character line-length target (#248). RFC 053 / #336 narrows the vertical-spacing contract, but it is not a general wrapping/configuration overhaul.
- `0.3` is still a development release. This page is intentionally incomplete until more post-`0.2` work lands.
