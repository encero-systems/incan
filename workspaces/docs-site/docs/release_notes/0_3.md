# Release 0.3

Incan 0.3 builds on the `0.2` line by making larger programs feel less improvised: richer source-level language features, a much broader standard library, a stronger test runner, better Rust interop, and fewer generated-Rust ownership surprises.

If `0.2` was mostly about explicit namespaces, library manifests, and Rust boundary cleanup, `0.3` is about using that structure for real application code. The release adds typed numerics, expression-oriented control flow, enum behavior, protocol hooks, Rust trait adoption, graph/collection/JSON/regex/datetime/logging/encoding/hash/compression stdlib modules, lazy iterator pipelines, `Result` combinators, first-class testing workflows, and tighter lockfile/formatter/tooling contracts.

If you are looking for the previous release story, start with [Release 0.2](0_2.md). For current user docs, start with [Control Flow](../language/explanation/control_flow.md), [Choosing numeric types](../language/how-to/choosing_numeric_types.md), [Testing in Incan](../language/how-to/testing_stdlib.md), the [standard library reference](../language/reference/stdlib/index.md), and [Rust interop](../language/how-to/rust_interop.md).

## What 0.3 is about

The main direction is not "more syntax for its own sake." `0.3` moves common project patterns into documented language and stdlib surfaces so users can write less Rust-shaped scaffolding and contributors can keep compiler behavior tied to explicit metadata.

- **Language**: Numeric widths, fixed-scale decimal annotations, `loop:` expressions, `if let` / `while let`, union narrowing, value enums, enum methods, computed properties, decorators, aliases, partial callables, protocol hooks, variadics, generators, and pattern alternation make the Python-shaped surface more expressive while staying statically checked.
- **Stdlib**: Collections, `OrdinalMap`, graphs, JSON values, regex, datetime, logging, encoding, hashing, compression, filesystem, I/O, temporary files, UUIDs, iterator adapters, and `Result` helpers move ordinary application needs out of ad hoc Rust interop.
- **Interop**: Rust crate imports, `rusttype`, trait adoption, associated types, derived Rust metadata, metadata-backed call boundaries, and generated Rust retention now cooperate better with real Rust crates and protobuf-style APIs.
- **Tooling**: `incan test`, `incan fmt`, `incan lock`, lifecycle commands, doctor diagnostics, checked API metadata, LSP metadata, and generated Rust audits are more deterministic and CI-friendly.
- **Architecture**: More behavior is registry- and metadata-driven, and generated Rust relies less on scattered special cases.

## Migrating from 0.2

Most ordinary `0.2` programs should continue to compile. The changes below are the ones most likely to show up during adoption.

1. **Formatter output may change.** `incan fmt` now follows RFC 053's vertical-spacing buckets and wraps more calls/signatures. Projects that run `incan fmt --check` should expect one intentional formatting diff.
2. **Numeric names are more reserved.** Existing `int` and `float` code keeps working, but project-local bare type names such as `decimal`, `numeric`, `bigint`, `integer`, `smallint`, `real`, or `double` can now collide with canonical numeric vocabulary. Rename local aliases or use the new exact forms such as `decimal[12, 2]`.
3. **Testing imports are clearer.** The language `assert` statement is always available, but testing decorators and helpers remain `std.testing` APIs. Files that use `@fixture`, `@parametrize`, `@skip`, `assert_eq`, or similar helpers should import them explicitly.
4. **Lockfiles are less noisy.** `incan.lock` no longer records generation timestamps, and routine `build` / `test` runs warn and reuse stale lock payloads instead of rewriting committed lockfiles. Run `incan lock` when you intentionally refresh the lock.
5. **New features are additive.** `loop:`, `if let`, `while let`, value enums, protocol hooks, iterator adapters, and `Result` combinators do not require rewriting existing `match`, `while True`, helper-function, or nested-`match` code.

## Features and Enhancements

### Language

- **Numeric types**: Numeric annotations cover exact integer widths, pointer-sized integers, `f32` / `f64`, analytics aliases, fixed-scale `decimal[p, s]` / `numeric[p, s]`, lossless widening, explicit resize helpers, and Rust boundary adaptation ([RFC 009], #325).
- **Control flow**: `loop:` can return values with `break <value>`, and single-pattern `if let` / `while let` bring the common match-and-bind pattern into the language ([RFC 016], [RFC 049], #327, #333, #387).
- **Pattern typing**: Pattern alternation, anonymous union annotations with narrowing, and value enums with raw `str` / `int` representations make tagged and externally encoded values easier to model ([RFC 071], [RFC 029], [RFC 032], #317).
- **Enums and object surfaces**: Enums can own methods and adopt traits, while models, classes, traits, and wrappers gained computed properties, protocol hooks, and operator hooks ([RFC 050], [RFC 046], [RFC 068], [RFC 028], #334, #203, #86, #162, #170).
- **Decorators and aliases**: Typed decorators, generic decorator factories, declaration aliases, RHS partial callable presets, variadic parameters, and call unpacking make callable composition more explicit ([RFC 036], [RFC 083], [RFC 084], [RFC 038], #437, #453, #83, #324, #640).
- **Iterators and generators**: Generator values and lazy iterator adapters make pipeline-style code possible without materializing intermediate lists ([RFC 006], [RFC 088], #127, #386).

### Rust Interop

- **Trait adoption**: Newtypes and rusttypes can adopt Rust traits with Incan source syntax and disambiguate same-name methods with `for Trait` ([RFC 043], #200).
- **Rust metadata**: Associated types, supported `@rust.derive(...)` metadata, and inspected Rust signatures now survive further through generated calls ([RFC 041], #175).

### Standard Library

- **`std.async`**: Async helpers cover `std.async.race`, channel reservation APIs, timeout-join helpers, cancellation-safe barrier behavior, and diagnostics for un-awaited async calls ([RFC 039], #173, #415, #416, #417, #418, #146).
- **`std.collections`**: Specialized containers cover ordered, sorted, counter, queue, stack, multimap, bidict, and import-free `list.repeat(value, count)` workflows ([RFC 030], [RFC 069], #385).
- **`std.collections.OrdinalMap`**: Deterministic immutable key-to-ordinal lookup supports schemas, catalogs, dictionary-encoded domains, and reproducible serialized lookup tables ([RFC 101]).
- **`std.compression`**: Codec-based compression covers gzip, zlib, deflate, bzip2, lzma, zstd, and Snappy-oriented byte/stream workflows ([RFC 061]).
- **`std.datetime`**: Temporal values cover date/time/datetime/duration construction, arithmetic, parsing, formatting, clocks, and timezone offsets ([RFC 058]).
- **`std.encoding`**: Binary-text transforms cover base64, hex, URL, and related byte/text conversion helpers ([RFC 064]).
- **`std.fs`**: Path-centric filesystem work covers paths, metadata, directory traversal, file operations, and Unix-like host behavior ([RFC 055]).
- **`std.graph`**: Graph primitives cover node/edge management, dependency ordering, traversals, path queries, and graph algorithms ([RFC 047]).
- **`std.hash`**: Hashing covers byte, file, and reader inputs with algorithm-specific helpers and normalized digest output ([RFC 065]).
- **`std.io`**: In-memory byte streams and buffered readers cover common byte-oriented I/O workflows without direct Rust interop ([RFC 056]).
- **`std.json`**: `JsonValue` supports dynamic JSON payload construction, inspection, conversion, and extraction at API boundaries ([RFC 051]).
- **`std.logging`**: Structured logging gives modules stable logger names, levels, fields, and runtime-friendly generated Rust output ([RFC 072]).
- **`std.regex`**: Safe-default regular expressions cover matching, captures, iteration, splitting, and replacement without opting into backtracking-only features ([RFC 059]).
- **`std.result`**: `Result[T, E]` gained Rust-shaped `map`, `map_err`, `and_then`, `or_else`, `inspect`, and `inspect_err` helpers ([RFC 070], #386).
- **`std.tempfile`**: Temporary file and directory helpers cover scoped temporary resources for test and application code ([RFC 010]).
- **`std.testing`**: Testing helpers and decorators cover fixtures, parametrization, markers, built-in temp/env fixtures, and async fixture support ([RFC 018], [RFC 019], [RFC 004], #76).
- **`std.uuid`**: UUID parsing, formatting, generation, version inspection, and byte/string conversion are available as first-party helpers ([RFC 060]).

### Tooling

- **`incan test`**: Inline `module tests:` blocks are discovered by the runner, and reporting supports resource-aware parallelism, JSON Lines, JUnit XML, durations, shuffling, and `--nocapture` ([RFC 018], [RFC 019], [RFC 004], #76).
- **`incan fmt`**: Formatting follows the vertical-spacing contract and wraps more long calls and signatures without changing language semantics ([RFC 053], #73).
- **Build policy**: `incan build`, `incan run`, and `incan test` support offline, locked, and frozen Cargo policy propagation ([RFC 020], #460).
- **Lifecycle commands**: `incan new`, `incan init`, `incan version`, `incan env`, and `incan tools doctor` cover project startup, environment inspection, offline readiness, and editor binary health ([RFC 015], #426).
- **Checked metadata**: Contract metadata flows through model bundles, project materialization, `.incnlib` artifacts, `incan tools metadata model`, `incan tools metadata api`, generated API docs, and LSP hover/command surfaces ([RFC 048], #205, #438).

## Bugfixes and Hardening

### Compiler Correctness

- **Argument planning**: Ownership and Rust-boundary coercion now route through shared argument-use and generated-use planning instead of parallel caller/emitter heuristics.
- **Duckborrowing**: Generated Rust now covers more typed use sites for arguments, returns, assignments, match scrutinees, collection elements, tuple elements, lookups, comprehensions, mutable aggregate parameters, Rust interop calls, and backend-inserted `Clone` bounds (#121, #241, #364, #366, #367, #372, #383, #391, #602).
- **Typed use sites**: InQL and release smoke testing fixed loop-item fields, union call arguments, storage-rooted match scrutinees, static list index assignment, typed `assert false` lowering, and const model metadata constructors (#620, #621, #622, #627, #630, #644, #671, #674).
- **Runtime expression fixes**: `std.regex` text borrowing, collection f-string formatting, and collection/string conversion diagnostics now fail less mysteriously at Rust or runtime boundaries (#624, #625, #71, #81).
- **Generic type flow**: Typechecking and lowering preserve more information for instantiated generic receivers, generic model/class field access, generic instance methods, list literals containing `Self`, trait/supertrait upcasts, imported prost oneof payloads, explicit call-site generic cycles, and locals initialized from static factory calls (#237, #231, #253, #230, #184, #218, #279, #252, #255).
- **Semantic guardrails**: Remaining semantic string comparisons in high-risk compiler paths are fingerprinted so new string-based behavior has to be centralized or explicitly classified.

### Rust Interop And Generated Rust

- **Borrowed Rust calls**: Borrowed `str`/bytes calls, method fallback borrowing, and retained generated imports now follow the same metadata-backed decisions across typechecking, lowering, and emission.
- **Rust bridge identity**: Rust bridge type identity is preserved across inspected methods with unknown generic or lifetime placeholders and across re-exported argument displays (#645, #630).
- **Rust collection adaptation**: Owned Incan values can flow to shared borrowed generic Rust parameters, and `list[T]` can adapt to `Vec<U>` where metadata proves the boundary (#506, #128).
- **Protobuf-style APIs**: Prost-style inherent and trait-provided `decode<T: Buf>(buf: T)` calls lower without user-authored workarounds (#609, #612).
- **Import retention**: Retained enum-pattern imports and metadata-derived extension-trait imports now survive generated Rust pruning (#459, #447).
- **Trait diagnostics**: Trait-typed local annotations now produce diagnostics instead of obscure lowering or generated-Rust failures (#462).
- **Generated manifest hygiene**: Tokio and `serde_json` stay behind feature gates, generated helper stubs use named helpers, and unused generated Rust is pruned without broad `allow` attributes (#351, #157, #214).

### Multi-File And Packages

- **Multi-file codegen**: Imported defaults expand with qualification, same-shaped union wrappers are shared through the crate root, wide union narrowing lowers fully, keyword-named modules escape consistently, and public submodule reexports work under `src/` (#395, #457, #461, #458, #122, #287).
- **Web retention**: Private route handlers and models are retained for web registration without making them user-visible public API (#117).
- **Public exports**: Public alias reexports, package-boundary alias consumption, lowercase exported static imports, imported static decorator strings, and keyword-named public symbols follow the same rules as ordinary library builds (#617, #631, #633, #658, #659).
- **Decorator metadata**: Checked API source signatures, imported/decorator `const str` arguments, generic decorator factory inference, and method-call decorator factories work across package boundaries (#636, #638, #640, #669).
- **Script/test manifests**: Generated script and test Cargo manifests include only reachable dependencies instead of blindly inheriting package-level heavy dependencies (#665).

### Formatter And Test Runner

- **Formatter/parser drift**: Tuple-unpack comprehensions and f-string debug markers now format without changing parse meaning (#615, #616).
- **Formatter retention**: Escaped f-string newlines, numeric literal spelling, qualified enum/constructor patterns, `mut` markers, block blank-line intent, docstrings, multiline trailing commas, long logical-expression wrapping, and class trait-adoption wrapping are preserved (#235, #250, #264, #289, #247, #394, #484, #565).
- **Comprehensions**: `?` propagation works correctly inside comprehensions, and collection/string conversion diagnostics are clearer when runtime coercion fails.
- **Inline tests**: Directory inline-test runs preserve each file's parser and import scope, conventional test batches split on imported-name collisions, and decorated functions named like builtins resolve to the source binding inside inline tests (#676, #677).
- **Test harness reuse**: `incan test` reuses more generated harness state, isolates single-file runs, keeps project cwd stable, and includes helper modules such as `std.result` when test files use helper-backed surfaces (#268, #269, #271, #288, #378, #610, #505).

### Docs And Dependencies

- **Docs**: User-facing docs were reorganized around reference/how-to/explanation boundaries for graph, regex, logging, hash, UUID, tempfile, collections, encoding, compression, datetime, and other stdlib modules (#284).
- **Contributor docs**: Contributor docs now describe crate boundaries, ownership metadata, staged Rust inspection, and the quarantined `std.web` host-runtime bridge (#284).
- **Dependency advisories**: Release hardening removes the `atomic-polyfill` advisory path through the local rust-analyzer proc-macro API patch, updates Wasmtime/WASI and MSRV together, and remediates Dependabot alerts across docs-site Python pins, VS Code lockfiles, Rust lock entries, and GitHub Actions (#260, #475, #464).
- **Docs-site pins**: `pymdown-extensions` is pinned to `10.21.3` for `GHSA-62q4-447f-wv8h` (#464).

## Known limitations

- Decimal arithmetic is not yet general language behavior. The `0.3` decimal surface covers typed annotations, literal validation, formatting, generated Rust representation, and display; arithmetic semantics need a follow-up language/library decision.
- `incan fmt` is still conservative. RFC 053 gives vertical spacing and common wrapping rules, but it is not a general pretty-printer overhaul for every nested expression shape.
- `std.regex` is a safe-default regular-expression surface, not a Python/PCRE compatibility layer. Lookaround, pattern backreferences, and other backtracking-only features belong in a separate package or future stdlib track if standardized.
- Native Windows filesystem behavior is not part of the `0.3` contract. `std.fs` documents Unix-like host behavior until the stdlib has an explicit platform split.
